  1. **`ROUND()` on double-precision values.** Root cause turned out to be much bigger than the one query that surfaced it: querying `information_schema.columns` showed virtually every rate/percentage column across ALL `savant_*` tables (`batting_avg`, `xwoba`, `hard_hit_percent`, `exit_velocity_avg`, `b_ab`, `b_total_hits`, etc. — 60+ columns) is stored as `double precision`, not `integer`/`numeric`, almost certainly because the ETL's pandas-based dynamic type inference (`create_table_if_not_exists` in `etl/update_savant_awsrds.py`) defaults a column to `float64` the moment any row has a NaN, which is common in Savant's leaderboard CSVs.
