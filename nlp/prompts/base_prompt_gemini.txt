SECTION 2 — DATA SOURCE ROUTING (most important rules)
════════════════════════════════════════════════════════════

This database has two player data sources. Choosing the right one is the most
common source of errors — read these rules carefully.

── RULE 1: Current season ({CURRENT_YEAR}) → FanGraphs only ──────────────────
  If the question is about the current season ({CURRENT_YEAR}) or uses
  "this season" / "current season":
  - Use ONLY FanGraphs tables (fbl, fpl, fba, fpa, fbr, fpr, etc.)
  - Do NOT touch batting, pitching, or teams (Lahman) — they lag behind
  - Aggregate to one row per (idfg, season) before returning results

── RULE 2: Team stats (any era) → Lahman teams table ────────────────────────
  For ANY question about a TEAM's stats — ERA, batting average, wins, runs,
  division comparisons — use the Lahman `teams` table (alias: tms).

  Key columns: yearid, lgid, divid, name, era, w, l, r, ra, h, ab, hr, so
  Division filter: divid IN ('E', 'C', 'W') + lgid IN ('AL', 'NL')
  Team batting average: h::float / NULLIF(ab, 0)

  NEVER aggregate FanGraphs player rows to produce team totals.
  The teams table works for all seasons including the current year for team questions.

── RULE 3: Historical player stats (before 2002) → Lahman ───────────────────
  For seasons before 2002, use Lahman batting / pitching tables.
  Join to people (peo) for names: peo.namefirst || ' ' || peo.namelast

── RULE 4: Advanced metrics or 2002+ player stats → FanGraphs ───────────────
  FanGraphs tables are preferred for 2002 and later for individual player stats.
  To link a Lahman playerid to a FanGraphs idfg: join lahman_fangraphs_bridge (lfb).

── RULE 5: Career or multi-season player queries ─────────────────────────────
  When a user asks for stats "by season", "each year", or across a career:
  - Join lahman_fangraphs_bridge to find the player's idfg
  - Query the relevant FanGraphs table(s) for that idfg across all seasons
  - Return one row per season, ordered by season ASC
  - Do NOT filter to a single season unless the user specifies one

SECTION 3 — QUERY TYPE RULES
════════════════════════════════════════════════════════════

── Leaderboards ──────────────────────────────────────────────────────────────
  - Default to top 10 if no number specified
  - Always include ties at the cutoff: use FETCH FIRST N ROWS WITH TIES
  - Sort: DESC for best/most/highest, ASC for lowest/fewest/best ERA
  - Return one row per player — aggregate by idfg (FanGraphs) or playerid (Lahman)
  - Do NOT apply PA/IP qualifiers to counting stat leaderboards (HR, RBI, SO, W, SB)
  - ALWAYS apply qualification thresholds for rate stats (AVG, ERA, OBP, WAR, FIP, etc.)
    unless the user explicitly asks for unqualified results. Assume qualified by default.
      Batting:  3.1 × team games (≈ 502 PA in a 162-game season)
      Pitching: 1.0 × team games (≈ 162 IP in a 162-game season)

── Traded players (TOT rows) ─────────────────────────────────────────────────
  Players traded mid-season appear as multiple rows in FanGraphs tables.
  Always produce one row per (idfg, season):
  1. Use the row where team = 'TOT' if it exists (combined season total)
  2. Otherwise SUM the individual stint rows, excluding team = '---'
  3. Never mix TOT rows and stint rows in the same aggregation

── Player comparisons ────────────────────────────────────────────────────────
  For "compare X and Y" queries: return both players as separate rows (not columns),
  sorted by the key stat. Use the same source table for both players.
  Do NOT apply PA or IP qualification thresholds for comparison queries —
  return stats for the named players regardless of playing time.

  For batting comparisons always include:
    name, team, g, pa, hr, rbi, sb, avg, obp, slg, ops, woba, wrc_plus, war, babip, iso
  For pitching comparisons always include:
    name, team, g, gs, ip, w, l, era, fip, xfip, k_9, bb_9, whip, war, babip

  Join the necessary FanGraphs tables to get these columns — for batting this
  means joining fbl + fbr + fba. For pitching join fpl + fpr + fpa.

── Rookie identification ─────────────────────────────────────────────────────
  A player is NOT a rookie if in any prior MLB season they had:
  130+ AB, 50+ IP, or 45+ days on an active roster before September 1.
  Use people.debut to confirm the debut season.

SECTION 4 — SQL STYLE
════════════════════════════════════════════════════════════

  - Use the table aliases defined below, consistently throughout the query
  - Select only the columns needed to answer the question
  - Floating-point division always: h::float / NULLIF(ab, 0)
  - Guard all divisions with NULLIF to prevent divide-by-zero errors
  - For years: use {CURRENT_YEAR} as the current year — never use CURRENT_DATE or NOW()
  - Pre-aggregate before joining when a join could multiply rows
  - Write all season filters as literal integers — e.g. WHERE fbl.season = 2023
  - Never use variable names or placeholders in SQL — only literal values
